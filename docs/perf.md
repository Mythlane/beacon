# Beacon — Performance & Overhead (v0.1)

## Hardware & Environment

Bench host: AMD Ryzen 9 7950X (16 cores, 32 logical, base 4.5 GHz),
127.1 GB DDR5, Windows 11 Pro 10.0.26200, JDK 25.0.1 LTS-27 (HotSpot 64-bit
Server VM, build 25.0.1+8-LTS-27). Desktop workstation, ambient air cooling,
relative-idle state during measurement (IDE foreground, no concurrent
stress workload).

These figures are a low-bound reference. Production servers running on
older or thermally-throttled hardware will see higher absolute ns/op
values, but the per-server %CPU stays comparable because the dominant
cost is constant work per world / per player, not contention or memory
bandwidth.

## Methodology

### Why synthetic JMH (and not end-to-end)

The end-to-end QUIC harness (`bench/src/main/java/.../OverheadHarness.java`,
`SessionSequence`) is implemented but blocked: it requires Hytale-protocol
packet IDs that are still provisional pending public release of the
Hytale server. Until that unblocks, end-to-end CPU/RSS deltas cannot be
measured against a real baseline.

The JMH benchmarks below sidestep that constraint: they measure the cost
of Beacon's own hot paths in isolation, with no dependency on the Hytale
SDK at runtime. The protocol layer is irrelevant to the per-call cost of
recording a tick, registering a player join, exporting a metric, or
submitting an export. End-to-end QUIC numbers will land in
`docs/perf-quic.md` once the live bench can run.

### JMH configuration

| Setting | Value |
|---|---|
| JMH version | 1.36 (plugin `me.champeau.jmh` 0.7.3) |
| Mode | `AverageTime` |
| Output unit | `ns/op` |
| Forks | 2 |
| Warmup | 3 iterations x 1s |
| Measurement | 5 iterations x 1s |
| Threads | 1 (single-threaded baseline) |

Total wall-clock for the 5 benchmarks: ~1m24s.

### Frequency assumptions

| Hot path | Frequency | Source |
|---|---|---|
| `recordOnce` | 1 Hz per world | `TpsMsptRecorder` polls each world once per second |
| `joinThenLeave` | 0.05 Hz per player | average 1 join+leave cycle every 20s per active player |
| `collectAll` | 0.033 Hz per server | `PeriodicMetricReader` flush every 30s |
| `submitWithSuccess` | 1 Hz per server | nominal export submission rate per OTLP batch |
| `submitWithFailure` | informational | excluded from projection (state-dependent, see Raw Results) |

## Raw Results

JMH 5-benchmark batch on the host above, results.json at
`bench/build/results/jmh/results.json`.

| Benchmark | ns/op | error (99.9% CI) | Frequency | ns/sec contributed (per server) | Notes |
|---|---:|---:|---:|---:|---|
| `TpsMsptRecorderBench.recordOnce` | 14.57 | +/- 0.86 | 1 Hz / world | 14.57 x worlds_per_server | hot path: cache lookup + histogram record + ConcurrentHashMap put |
| `PlayerCountRegistryBench.joinThenLeave` | 25.98 | +/- 2.28 | 0.05 Hz / player | 1.30 x players_per_server | full join+leave cycle on a 100-player ring buffer |
| `HytaleMetricsObserverBench.collectAll` | 233.96 | +/- 15.01 | 0.033 Hz / server | 7.80 (constant) | invokes both gauge callbacks via `InMemoryMetricReader.collectAllMetrics()` |
| `ExportFailureHandlerBench.submitWithSuccess` | 8.43 | +/- 0.41 | 1 Hz / server | 8.43 (constant) | nominal `submit()` + `onSuccess()` cycle, breaker CLOSED |
| `ExportFailureHandlerBench.submitWithFailure` | 1.68 | +/- 0.04 | n/a | excluded | breaker trips OPEN after 5 failures; subsequent calls short-circuit. Informational only. |

Interpretation of `submitWithFailure`: the 1.68 ns/op is the cost of the
breaker-OPEN short-circuit path, not the cost of failure handling. In a
real degraded state (OTLP backend down), this is exactly what runs at
each `submit()` call. The number is included for completeness; it is not
projected because its real-world frequency depends on backend health, not
on Beacon's own cadence.

## Multi-Scenario Projection

Generated by `./gradlew :bench:overheadReport` from `results.json`.

```
Scenario  Topology                          per-server %CPU   fleet (cores)  Verdict
small     1 srv x 4 wld x 50 pl                   0.0014%       0.000000   PASS
medium    10 srv x 40 wld x 500 pl                0.0014%       0.000001   PASS
large     50 srv x 200 wld x 5000 pl              0.0020%       0.000010   PASS
```

Verdict thresholds (per-server %CPU):

| Scenario | PASS | WARN | FAIL |
|---|---|---|---|
| small | < 0.1% | 0.1-0.5% | >= 0.5% |
| medium | < 0.3% | 0.3-0.7% | >= 0.7% |
| large | < 0.5% | 0.5-1.0% | >= 1.0% |

Top per-server contributors at `large`: `joinThenLeave` 64%,
`recordOnce` 29%. The `joinThenLeave` share grows with players-per-server
(small/medium have 50, large has 100), which is expected: it is the only
projected hot path that scales with player count rather than world count.

The per-server %CPU is roughly constant across the three scenarios. This
is the property that matters for fleet operators: adding servers does not
amplify Beacon's overhead on any single node. Each server pays its own
local cost; cross-fleet aggregation happens in the OTel backend, not in
the plugin.

## Verdict v0.1

Per-server overhead is stable at ~0.0014-0.0021% across all three
projected topologies. Against the v0.1 release-blocker target of < 1%
CPU at 20 players (`CLAUDE.md`, requirement 1.10), the measured headroom
is over 500x.

Fleet aggregate at `large` (50 servers x 200 worlds x 5000 players)
sums to 0.000010 cores - a single physical core covers Beacon's work
for 100,000 such fleets.

Release criterion 1.10 is met with the JMH harness; the QUIC end-to-end
gate (1% hard fail, 1.5% noise headroom) remains gated on Hytale public
release and will be re-validated in `docs/perf-quic.md` when unblocked.

## Limitations

Three blind spots that the JMH harness does not cover. Be honest about
them when citing these results.

1. **Allocation pressure over time.** JMH measures one operation at a
   time; it does not capture the cumulative effect of 1+ hour of steady
   allocation on GC pause distribution, old-gen growth, or Metaspace
   fragmentation. A hot path that allocates 64 bytes/op looks fine at
   the operation level and can still trigger young-gen pressure at
   sustained load. To close: run a 4-hour soak with allocation profiler
   (async-profiler `-e alloc`) on the live QUIC bench.
2. **Multi-thread contention.** All five benchmarks run with `@Threads(1)`
   and `fork=2` (sequential, not concurrent). Real Beacon paths cross
   threads: `playerJoined`/`playerLeft` are invoked from network event
   loops, gauge callbacks run on the OTel reader thread, the tick poller
   runs on its own scheduler. The `ConcurrentHashMap` operations
   measured here are uncontended. Under contention, the cost can grow
   non-linearly. To close: add `@Threads(8)` variants on
   `PlayerCountRegistryBench` and `TpsMsptRecorderBench` (Phase 4).
3. **Network effects.** `submitWithSuccess` measures a no-op cycle
   (`submit()` + `onSuccess()`). It does not include OTLP serialization,
   gRPC frame construction, or backend latency / backpressure. A real
   OTLP exporter against a slow or throttling backend will queue,
   retry, and eventually trip the breaker - costs that the synthetic
   bench cannot reach. To close: end-to-end overhead bench with a real
   OTLP collector (in `docs/perf-quic.md`).

## Reproducing these benchmarks

```bash
./gradlew :bench:jmh                         # ~1m24s, regenerates results.json
./gradlew :bench:overheadReport              # full bench + projection
./gradlew :bench:overheadReport -PskipJmh    # re-project from cached results.json
```

The `overheadReport` task fails the build (`GradleException`) if any
scenario crosses its FAIL threshold, so it can be wired into CI as a
regression gate.
